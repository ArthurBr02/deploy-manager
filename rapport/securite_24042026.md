# Rapport d'Analyse de Sécurité - 24/04/2026

## Introduction
Ce rapport présente une analyse de sécurité approfondie de l'application **Deploy Manager**, focalisée sur trois axes majeurs : la prévention des injections de commandes, la configuration de la sécurité réseau (CORS/CSRF) et la validation des données (DTO).

---

## 1. Analyse des Injections de Commandes (RCE)
**Statut : 🔴 CRITIQUE**

### Observations
Le système de déploiement utilise un mécanisme de remplacement de variables dans `DeploymentService.java` :
```java
private String replaceVariables(String command, Host host) {
    return command
            .replace("{host}", host.getName() != null ? host.getName() : "")
            .replace("{ip}", host.getIp() != null ? host.getIp() : "")
            .replace("{domain}", host.getDomain() != null ? host.getDomain() : "");
}
```
Ces variables sont ensuite passées à un `ProcessBuilder` via un shell (`sh -c` ou `cmd /c`).

### Risques identifiés
*   **Absence de sanitisation :** Les champs `name`, `ip` et `domain` de l'hôte ne sont pas filtrés. Un administrateur malveillant ou un compte compromis peut injecter des commandes shell.
*   **Exemple d'attaque :** Si un hôte a pour domaine `example.com; rm -rf /`, la commande finale exécutée sera `sh -c "votre_commande example.com; rm -rf /"`.

### Recommandations
1.  **Validation stricte au niveau DTO :** Utiliser des Regex pour valider les IP et les noms de domaine.
2.  **Échappement des caractères shell :** Utiliser une bibliothèque pour échapper les arguments ou interdire les caractères métashell (`&`, `;`, `|`, `$`, `>`, `<`, etc.) dans les variables de remplacement.

## 2. Configuration CORS et CSRF
**Statut : 🟡 MODÉRÉ**

### Observations
Dans `SecurityConfig.java` et `AuthService.java` :
*   **CSRF :** Désactivé via `csrf(AbstractHttpConfigurer::disable)`.
*   **CORS :** Configuration via `setAllowedOriginPatterns` autorisant `http://localhost:*`.
*   **Cookies :** Le Refresh Token est stocké dans un cookie `HttpOnly`, mais le flag `Secure` est forcé à `false` dans le code.

### Analyse
*   **CSRF :** Puisque l'authentification est basée sur un Header `Authorization: Bearer <token>` et non sur des cookies automatiques pour l'Access Token, la désactivation du CSRF est acceptable.
*   **CORS :** L'utilisation de `http://localhost:*` avec `setAllowCredentials(true)` est trop permissive pour un environnement de production.
*   **Sécurité des Cookies :** L'absence du flag `Secure` sur le cookie de refresh permet son interception si l'application est utilisée en HTTP (non-TLS), ce qui est risqué même si le token est HttpOnly.

### Recommandations
1.  **Restreindre les Origins :** En production, spécifier l'URL exacte du frontend (ex: `https://deploy.mondomaine.com`).
2.  **Flag Secure Dynamique :** Rendre le flag `Secure` configurable via `application.yml` pour qu'il soit à `true` en production.
3.  **Secrets par défaut :** S'assurer que les secrets JWT (`JWT_ACCESS_SECRET`, etc.) sont impérativement surchargés par des variables d'environnement en production, car les valeurs par défaut sont présentes dans le code source.
4.  **Sécurité des Tokens SSE :** Les tokens passés en paramètre d'URL (pour EventSource) peuvent être loggués par les serveurs proxies. Privilégier des tokens à usage unique et courte durée de vie pour les connexions SSE.

## 3. Validation des DTO et Intégrité des Données
**Statut : 🟠 IMPORTANT**

### Observations
Les DTO (ex: `HostRequest.java`, `UpdateUserRequest.java`) utilisent des validations minimales.
```java
public record HostRequest(
    @NotBlank String name,
    @NotBlank String ip,
    @NotBlank String domain,
    // ...
) {}
```

### Risques identifiés
*   **Format de données :** L'absence de validation `@Pattern` permet l'insertion de données invalides ou malveillantes dans la base de données.
*   **Déni de Service (DoS) :** Bien que l'upload d'avatar soit limité à 10 Mo et compressé, l'utilisation de `ImageIO.read` sur des fichiers non validés peut exposer à des attaques de type "Image Bomb".

### Recommandations
1.  **Validation de Format :** Ajouter des contraintes Bean Validation :
    *   `@IPv4` ou `@Pattern` pour les IP.
    *   `@Size` pour limiter la longueur des champs et éviter les saturations de base de données.
2.  **Validation des Commandes :** Implémenter une liste blanche de commandes autorisées ou un validateur de syntaxe shell pour les commandes de déploiement définies par les admins.

---

## Conclusion
L'application présente une base solide en termes de gestion des droits (RBAC) et d'authentification (Argon2, JWT). Cependant, la **vulnérabilité aux injections shell est une priorité absolue** qui doit être corrigée avant toute exposition sur un réseau non sécurisé. La mise en place de validations strictes sur les entrées utilisateurs est essentielle pour garantir l'intégrité globale du système.
